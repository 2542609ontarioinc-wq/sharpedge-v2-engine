Soccer Pipeline Fix — June 22, 2026
=====================================

ROOT CAUSE
----------
Two compounding bugs caused the pipeline to silently produce 0 picks even
when the Odds API returns coverage. GitHub also dropped the 6 AM UTC scheduled
run on June 22 (no June 22 run appears in gh run list), but even if it had
fired, the code bugs below would have produced 0 picks.

Bug 1 — sync_soccer_odds: INSERT instead of delete-then-insert
  Every pipeline run appended new odds rows without clearing the previous
  day's rows. After 6 daily runs, soccer_odds had 1,144 rows with
  game_id=NULL. Newly inserted rows from the current run landed at the
  bottom of that pile.

Bug 2 — match_odds_to_games: hard LIMIT 1000 on both queries
  The script fetched unmatched odds with LIMIT 1000 (no ordering) and
  all soccer games with LIMIT 1000. The games table had 2,152 rows, so
  today's freshly synced fixtures were beyond the cutoff. The matcher
  scanned only stale historical rows and matched 0. Affected games today:
    - IK Oddevold vs Ljungskile SK (name: "IK Oddevold" vs "Oddevold")
    - Shelbourne Dublin vs Bohemians (name: "Shelbourne Dublin" vs "Shelbourne")
    - Varbergs BoIS vs Landskrona BoIS (matched at 0.95 once fixed)
    - Örebro SK vs Sandvikens IF (unicode: "Örebro" vs "Orebro")
    - Ponte Preta vs Grêmio Novorizontino (suffix mismatch, matched at 0.89)


FIXES APPLIED
-------------
1. src/ingestion/sync_soccer_odds.py
   Before inserting, delete all soccer_odds rows where
   captured_at >= today UTC. This makes every run idempotent:
   re-running the pipeline on the same day replaces that day's odds
   rather than accumulating duplicates. Historical rows (prior days)
   are untouched.

2. src/ingestion/match_odds_to_games.py
   - Odds query: added .gte('captured_at', 24h_ago) so only fresh rows
     are processed. Stale unmatched rows from prior days are ignored.
   - Games query: replaced LIMIT 1000 with .gte('game_date', today)
     .lte('game_date', today+7d) — only upcoming games are candidates,
     so the query stays well under any row limit as the table grows.

3. .github/workflows/soccer_daily_automation.yml
   Cron shifted from "0 6 * * *" to "0 10 * * *" (10 AM UTC = 6 AM
   Toronto). The 6 AM UTC slot was running 6-10 hours late on June 19-21
   and missed entirely on June 22. 10 AM UTC is a less congested slot.

4. CLAUDE.md
   Removed stale note claiming the workflow calls run_full_soccer_engine.
   It was updated to run_sharp_engine in commit 11e8053. Corrected to
   reflect the actual state.
