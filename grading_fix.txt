WHAT WAS FIXED
--------------
Three bugs closed. No soccer or prediction logic touched.

1. "post" not caught as postponed  [grade_mlb_picks.py + grade_mlb_model_picks.py]
   API-Sports sets status=POST for postponed games. "post" was absent from
   POSTPONED_STATUSES, so a POST game with period=null/empty could fall through
   _is_finished() as unclassified — neither finished nor postponed — and if the
   games table row briefly had period="Finished" due to a data timing issue, it
   would be treated as finished. "post" is now in POSTPONED_STATUSES in all
   three grading files, so status=POST always short-circuits to False regardless
   of what period says.

   Also fixed in grade_mlb_prop_picks.py: "post" was incorrectly listed in
   FINISHED_STATUSES there (would have caused postponed games to be treated as
   finished for prop grading). Removed it and added a proper POSTPONED check.

2. 0-0 placeholder score graded as real result  [grade_mlb_picks.py + grade_mlb_model_picks.py]
   A 0-0 final is impossible in MLB. If a game somehow passes the status check
   with home_score=0, away_score=0 (placeholder), the grader now skips it and
   leaves the pick pending. This is the direct cause of the "Atlanta Braves +2.5
   | actual 0-0 | WIN" bad grade: 0 + 2.5 > 0 → WIN on a game that never ran.

   Sharp picks loop and safe zone loop both have the guard:
     if hs == 0 and as_ == 0: continue

3. game_date silently stripped on every upsert  [grade_mlb_picks.py]
   _upsert_grade() had a try/except that caught column-not-found errors for
   game_date and retried without it. This meant every graded row had game_date=NULL
   in mlb_pick_grades even though the games table has the date. The fallback is
   removed; the column is now always written. Requires sql/087 to be applied first
   (see SQL section below).

   grade_mlb_model_picks.py also now writes game_date to mlb_model_grades.
   Requires sql/088 to be applied first.
