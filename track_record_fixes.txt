FIX 1 — CLV DISPLAY GATING (most important)
--------------------------------------------
Problem:
  SegmentStats rendered seg.avgClv.toFixed(2)% with no sample count, so a
  single-digit CLV sample averaging near zero displayed as "+0.00%" — identical
  in appearance to a mature 40-pick "+0.00%" result. The near-zero value was
  real (not a fabricated 0-default), but the missing context made it misleading.

Root cause confirmed:
  _num(None) in build_mlb_pick_detail.py returns None (not 0), so picks with
  no CLV snapshot are correctly excluded. Near-zero average came from a tiny
  real CLV sample in mlb_clv_tracking.

Changes:
  types.ts — Added clvSampleCount: number to MLBSubscriberSegment.
             (Comment in deprecated rowToSegment() sets it to 0 since
             mlb_subscriber_results has no clv_sample_count column.)

  MLBSubscriberView.tsx — Added constant:
    const MIN_CLV_SAMPLE = 10;

  computeSegment() — Added to return object:
    clvSampleCount: clvRows.length,

  SegmentStats — Replaced bare avgClv.toFixed(2)% with:
    if clvSampleCount >= 10 → "+X.XX% (n=X)"
    if clvSampleCount < 10  → "— (n=X, building)"

Display examples:
  Before: "+0.00%" (no context; looks authoritative at any n)
  After:  "— (n=3, building)"  when n < 10
  After:  "+0.04% (n=12)"      when n >= 10


FIX 2 — CLV BEAT RATE RENDERED IN SegmentStats
-----------------------------------------------
Problem:
  computeSegment() already computed clvBeatRate correctly:
    clvRows.filter((p) => p.beatClose).length / clvRows.length
  But SegmentStats never rendered it — the metric was computed and dropped.
  Beat rate is a more honest signal than avg CLV at low n because it is less
  sensitive to outliers.

Changes:
  MLBSubscriberView.tsx — Added CLV Beat Rate StatCell to SegmentStats:
    const beatRate = clvN >= MIN_CLV_SAMPLE && seg.clvBeatRate != null
      ? `${(seg.clvBeatRate * 100).toFixed(0)}% (n=${clvN})`
      : `— (n=${clvN}, building)`;
    <StatCell label="CLV Beat Rate" value={beatRate} />

  SegmentStats grid is now 7 cells (was 6).
  Same MIN_CLV_SAMPLE gating applies to beat rate and avg CLV.

Display examples:
  "— (n=3, building)"  when CLV sample < 10
  "58% (n=12)"         when CLV sample >= 10

FIX 4 — STEP-11 LOUD FAILURE
------------------------------
Problem:
  build_mlb_pick_detail.py (step 11) printed "✅ MLB pick detail built: N rows"
  but never asserted N > 0. If a Supabase upsert error caused 0 rows to be
  written while grades existed, the step would appear to succeed and the engine
  would continue silently, leaving mlb_pick_detail stale. The track record would
  then freeze at the last successful run's numbers with no visible error.

Changes:
  build_mlb_pick_detail.py — Added import sys at the top.
  Added after the success print:
    if saved == 0 and len(grades) > 0:
        sys.exit(
            f"❌ step-11 failure: {len(grades)} graded picks exist but 0 rows "
            "written to mlb_pick_detail — check upsert errors above"
        )

  sys.exit() with a string message causes run_mlb_engine.py's subprocess call
  to return a non-zero exit code, which will log the step as failed and (if
  run in CI or a supervised job) surface the error visibly.
