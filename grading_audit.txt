ROOT CAUSE OF THE 0-0 / WRONG-GRADE BUG
-----------------------------------------
The grader (src/grading/grade_mlb_picks.py) trusts the `games` table status
fields and NEVER checks whether the score is real before grading.

The SF Giants @ Atlanta Braves game on 2026-06-18 was postponed and sits in
the `games` table with:
  status=POST | period=Postponed | home_score=0 | away_score=0 (placeholder)

The bug fired when the grader ran at a moment when `period` was not yet set to
"Postponed". At that point:
  - status = "post" → NOT in POSTPONED_STATUSES (has "postponed" but not "post")
  - period = null/empty → also not caught
  - _is_finished() fell through and returned True (incorrectly treated as finished)

With 0-0 score, grading "Atlanta Braves +2.5" (safe_banker):
  team_margin = home - away = 0 - 0 = 0
  team_margin + line = 0 + 2.5 = 2.5 > 0  →  WIN  (wrong — game never played)


THREE CONCRETE BUGS IDENTIFIED
-------------------------------
1. "post" not in POSTPONED_STATUSES
   File: src/grading/grade_mlb_picks.py:28
   Effect: status=POST games only protected if period="Postponed" is also set.
           If period is null/empty, the game falls through as unclassified.
           Fix: add "post" to POSTPONED_STATUSES.

2. No guard against 0-0 scores being graded
   File: src/grading/grade_mlb_picks.py:main()
   Effect: any game with placeholder 0-0 score and a status that looks
           "finished" will be graded. In MLB a 0-0 final is impossible.
           Fix: skip grading if home_score == 0 and away_score == 0.

3. game_date column missing from mlb_pick_grades (sql/087 not applied)
   File: src/grading/grade_mlb_picks.py:189-196 (_upsert_grade fallback)
   Effect: all 51 graded rows have game_date=NULL; the fallback silently
           strips the field on every upsert.
           Fix: run sql/087 migration to add game_date column, then remove
           the try/except fallback.
