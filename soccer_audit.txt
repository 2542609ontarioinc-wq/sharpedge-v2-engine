━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

data.ts getSharpPicks() line ~148:
  const isRealValue = hasOdds && edge !== null && edge > 0 && edge <= 15;

Today's Colombia vs Congo DR pick:
  model_edge = 22.86%  →  isRealValue = FALSE
  The "edge" badge does NOT render on this pick.

Today's England vs Ghana pick:
  model_edge = 10.14%  →  isRealValue = TRUE  ✅

The engine published the Colombia pick as "Elite Value" with final_allowed=true.
The frontend's independent edge cap (≤15) overrides the engine's judgment and
strips the value badge. WC picks against heavy underdogs routinely produce edges
20-30% because the engine sees large expected-goal gaps. The cap was appropriate
for league soccer where edges are smaller; it's too tight for the WC slate.

The pick still SHOWS (isRealValue doesn't gate visibility), but the edge badge
is missing, making it look like a no-edge pick.

FIX: Raise cap to 30% (matching the engine's max edge band) OR replace the
frontend's own edge cap with trust in the engine's final_allowed=true + the
engine's own edge_flag check in apply_honest_calibration.py (-10 ≤ edge ≤ 15
is the engine's sane-band). If the engine approved it, the frontend should show it.

5. SAFE ZONE ROI BUG
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

soccer_pick_results currently shows:
  safe_balanced: W=12, L=6, ROI=-50.0%  (12 graded rows: 6 WIN, 6 LOSS)
  safe_banker:   W=4,  L=2, ROI=-33.33%

Root cause:
  grade_soccer_results.py line 47:
    total_units = sum(float(p.get("units_result") or 0) for p in graded)
    roi = round(total_units / len(graded) * 100, 2)

  soccer_pick_grades_v2 has units_result=0.0 for WIN and units_result=-1.0
  for LOSS on safe zone picks. Safe zone picks have no specific odds at pick
  time (balanced/banker are categorical, not priced picks). Treating losses as
  -1 unit and wins as 0 produces a grossly negative ROI that has no meaning.

  The TrackRecordView displays these numbers to users. -50% ROI from a no-odds
  safe zone product looks like a massive failure when it's just a calculation
  artifact.

FIX option A: Exclude ROI column from safe zone market rows in the track record
  display (TrackRecordView.tsx — mark safe_balanced/safe_banker ROI as "N/A — no odds").
FIX option B: Fix the units_result assignment in grade_soccer_results.py to
  use a fixed 1.0 baseline for safe zone picks (match the safe zone's actual
  intent — these are flat-stake recommendations, not edge-sized bets).
Files: src/models/grade_soccer_results.py,
       web/src/components/TrackRecordView.tsx

getSharpPicks() in data.ts fetches ALL rows from final_pro_soccer_picks with
no date filter of its own.

The engine DOES handle this:
  build_final_pro_soccer_picks.py line 162:
    supabase.table("final_pro_soccer_picks").delete().neq(...)
  (deletes all current rows, then re-inserts only games with date >= toronto today)

So picks are cleaned on every engine run. This is fine when the engine runs daily.

yesterday's picks persist in the table and the frontend shows them.

the engine had not yet run for June 23.

independent guard (same pattern getSafeZone() already uses via the games join).
This makes the UI resilient to engine-skip days without depending on cleanup.

    back to a canonical per-pick record

This is prerequisite infrastructure for a soccer subscriber view with real stats.
This is planned work, not a quick fix.
